Desafío 01

SQL: El límite del esquema rígido

Al intentar insertar un tercer lote incluyendo un quinto valor para la "Humedad" (ej: 40.5) sin declarar la columna previamente en el esquema, se ejecuta:

Python (SQLite) 
cursor.execute("INSERT INTO PRODUCCION VALUES (3, 'Pilsner', 250, 12.0, 40.5)")
Error devuelto por Python / SQLite
sqlite3.OperationalError: table PRODUCCION has 4 columns but 5 values were supplied

¿Por qué SQL no permite esto sin un ALTER TABLE?
Las bases de datos relacionales tradicionales como SQL son Schema-on-Write (Esquema a la escritura). Esto significa que la base de datos es un contrato estricto: antes de aceptar cualquier dato, su estructura (columnas, tipos de datos) debe estar explícitamente definida. No admite datos sorpresa para mantener la integridad relacional.

Solución ejecutada: Modificar la estructura de la tabla (Data Definition Language) antes de insertar el dato.

Solución SQL 
# 1. Agregar la columna a la estructura
cursor.execute("ALTER TABLE PRODUCCION ADD COLUMN humedad REAL")

# 2. Insertar el dato (ahora sí concuerdan los valores con las columnas)
cursor.execute("INSERT INTO PRODUCCION VALUES (3, 'Pilsner', 250, 12.0, 40.5)")
conn.commit()
Resultado
Lote ingresado exitosamente con columna humedad soportada.
Desafío 02

NoSQL: La Variedad y la ausencia de límites

Creación de doc3 con propiedades experimentales totalmente disruptivas, sin precedentes en doc1 ni doc2:

Python (Simulación NoSQL) 
doc3 = {
    "id": 3,
    "estilo": "Cerveza Experimental",
    "ingrediente_secreto": "Miel de Ulmo",
    "ph_nivel": 4.5,
    "vibracion_sonora": "Mozart"
}

base_nosql.append(doc3)
print(json.dumps(base_nosql, indent=2))

Al ejecutar y consultar, el diccionario/JSON se inyecta e imprime sin lanzar ningún error ni advertencia, conviviendo pacíficamente en el mismo dataset que los demás lotes.

¿Tuvieron que pedirle "permiso" a la base de datos para guardar estos datos locos?
No. Absolutamente ningún permiso.

Fundamento: Los sistemas NoSQL basados en documentos (como MongoDB, modelados aquí mediante diccionarios JSON) operan bajo arquitectura Schema-less o de esquema dinámico. La validación estructural se delega generalmente a la aplicación (Schema-on-Read), no a la base de datos.

Por lo tanto la base de datos asume que todo documento JSON bien formado es válido per se, proporcionando una adaptabilidad líquida (Variedad en Big Data) que es vital cuando los atributos del negocio cambian repentinamente, como en esta cerveza experimental.
