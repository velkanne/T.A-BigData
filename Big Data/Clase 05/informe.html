Pregunta 01

Integridad: Rechazo de temperatura inválida (45 °C)

La tabla PRODUCCION fue creada con una restricción CHECK que limita el campo temperatura_fermentacion al rango de 0 a 30 °C:

SQL 
CREATE TABLE IF NOT EXISTS PRODUCCION (
    id_lote     INTEGER PRIMARY KEY,
    estilo      TEXT NOT NULL,
    litros      INTEGER,
    temperatura_fermentacion REAL CHECK(temperatura_fermentacion BETWEEN 0 AND 30)
)

Cuando se intentó insertar un lote con temperatura de 45.0 °C, SQLite rechazó la operación al violar la restricción CHECK:

Python 
cursor.execute("INSERT INTO PRODUCCION VALUES (2, 'Stout', 300, 45.0)")
Salida — Error capturado
Bloqueo SQL esperado: CHECK constraint failed:
temperatura_fermentacion BETWEEN 0 AND 30
(La restriccion CHECK funciono)

El registro nunca se insertó. La base de datos actuó como guardián: impidió que un dato fuera de rango contamine la tabla.

Relación con la Veracidad: La Veracidad exige que los datos almacenados sean confiables y representen fielmente la realidad. Una temperatura de 45 °C es fisicamente invalida para la fermentación de cerveza (las levaduras mueren sobre los ~35 °C). La restricción CHECK garantiza que solo ingresen datos veraces al sistema, rechazando automáticamente valores absurdos antes de que corrompan análisis posteriores.

Pregunta 02

Flexibilidad: Esquema dinámico en NoSQL

No fue necesario modificar ninguna estructura previa. La simulación NoSQL (diccionarios Python + JSON) permitió insertar dos documentos con esquemas completamente distintos sin declarar columnas ni alterar tablas:

// Documento 1 — campos básicos
{
  "id": 1,
  "estilo": "IPA",
  "detalles": { "temperatura": 18.5, "operario": "Francisco" }
}

// Documento 2 — campos adicionales sin migración
{
  "id": 2,
  "estilo": "Stout",
  "iot_sensor_data": [18.2, 18.5, 19.0, 18, 4],
  "clima_exterior": "Lluvia en Osorno",
  "alertas": "Nivel de espuma alto",
  "maestro_cervecero": "Analista TP"
}

El segundo documento incluye iot_sensor_data, clima_exterior, alertas y maestro_cervecero — campos que no existen en el primero. En SQL, esto habria requerido ejecutar ALTER TABLE para agregar cada nueva columna.

Relación con la Veracidad: En Big Data, las fuentes de información cambian constantemente (sensores IoT, clima, alertas en tiempo real). La flexibilidad del esquema NoSQL permite capturar toda la información disponible sin descartarla por no encajar en una estructura rígida. Esto aumenta la Veracidad porque se preserva el contexto completo del dato, incluyendo variables ambientales y operativas que un esquema fijo obligaría a omitir o forzar a formatos inadecuados.

Pregunta 03

Decisión de Arquitectura: Facturación vs. Redes Sociales

Facturación → SQL

  • Datos estructurados y predecibles (RUT, monto, fecha, folio)
  • Requiere integridad referencial estricta (clientes, productos, impuestos)
  • Transacciones ACID obligatorias para cumplimiento legal y tributario
  • Restricciones CHECK y FOREIGN KEY previenen datos corruptos
  • Auditoría y trazabilidad exigidas por normativa (SII)

Redes Sociales → NoSQL

  • Datos semi/no-estructurados (texto, imágenes, emojis, hashtags)
  • Cada comentario puede tener campos distintos (likes, replies, media)
  • Alto volumen e ingesta en tiempo real
  • Escalabilidad horizontal prioritaria sobre consistencia estricta
  • El esquema evoluciona constantemente sin migraciones

Criterio de decisión: La facturación exige que cada registro sea correcto, íntegro y legalmente válido — las restricciones SQL garantizan eso (como se demostró con el CHECK en la temperatura). Los comentarios en redes sociales priorizan capturar el mayor volumen y variedad posible sin perder datos por rigidez de esquema — exactamente lo que demostró la simulación NoSQL al aceptar documentos con estructuras diferentes.
